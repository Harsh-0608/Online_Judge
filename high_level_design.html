High-Level Design (HLD): Premium Online Judge Platform

1. System Overview

The platform is a secure, high-performance online judge system that enables users to solve coding challenges, run code against custom inputs, and participate in contests. The application features a glassmorphic dark-mode frontend, an integrated Monaco code editor, real-time code evaluation sandboxing, and a bounded concurrency task queue designed for optimal host server stability.

4. Core Technical Subsystems

A. Concurrently Bounded Task Execution Queue

The code execution utility in sandbox.js maintains an in-memory queue that restricts peak execution concurrency to avoid crashing the server. Key details include:

  • Queue Array Structure: Stores functions and their promise resolvers in a FIFO structure: queue.push({ fn, resolve }).
  • Concurrency Limiting: Governed by MAX_CONCURRENT_RUNS, defaulting to the host CPU cores minus 1: Math.max(1, os.cpus().length - 1).
  • Isolated Container Execution: Spawns compilers (Python, Node, G++, Java) in temporary read-only folders inside the temp_runs folder.
  • Self-Cleaning: Clears folders immediately after run execution. Any orphaned folders are recursively wiped when the Express gateway starts up in server.js to prevent disk capacity exhaustion.

B. Session Authentication & Security

  • Authentication Flow: Handled by AuthContext.jsx on the frontend and auth.js middleware on the backend using JWT (JSON Web Tokens). Tokens are stored in browser localStorage and passed in requests using the Authorization: Bearer <token> header.
  • Guest Mode isolation: Clicking "Continue as Guest" signs in a shared guest account. To prevent guests from seeing each other's submissions, the server automatically resets the guest account's submission history and solved count to zero every time a guest logs in.
  • Security Constraints: Executing code containers are stripped of network access (--network none) to prevent data exfiltration, memory-bound to 256MB, and time-capped (e.g. 2000ms) to prevent infinite loops and resource hogging.

5. Features Built & Integrated

  • Glassmorphic Pagination: Custom glass-card navigation controls built for the Problems listing and User Profile histories, avoiding large payload lags.
  • Contest Workspace Mode: Users solving problems from a contest are restricted within the contest workflow (back-routing leads back to the active contest dashboard rather than the general problem page).
  • Contest Timer: Embedded active countdown clocks in the workspaces that warn users and lock submissions once the contest expires.
  • Dynamic Contest Statuses: Contest statuses (Upcoming, Active, Completed) are calculated on-the-fly on backend fetches according to current server timestamps, keeping listing badges in perfect synchronization.
  • Diagnostic Sandbox Results: Displays descriptive compiler error stack traces, runtime faults, or Time Limit Exceeded (TLE) messages on dry runs and code submits.

  • Frontend Hosting: Deployed as a static React bundle on **Vercel** with automatic HTTPS.
  • Automatic API Fallback Routing: The client-side configuration checks if the user is visiting from a *.vercel.app hostname. If true, it automatically routes all API traffic directly to the secure EC2 backend.
  • Backend Server: Orchestrated using **Docker Compose** on an **AWS EC2** instance.
  • Static Networking (Elastic IP): Associated an AWS Elastic IP to keep the server IP static, avoiding broken API domain mapping endpoints.
  • Nginx Reverse Proxy & SSL: Handles incoming API routes on port 443 and proxies traffic to the Express container. Features secure TLS/SSL encryption generated via Certbot Let's Encrypt certificates.

7. Technical Challenges & Solutions

Challenge / Bug Encountered Root Cause Analysis Resolution Applied
Server Crashes on Login / Submissions Docker overlay layers and compile files filled up the AWS EC2 instance disk (6.7GB root volume), resulting in No space left on device database write failures. Performed deep system vacuuming, systemctl containerd cache resets, and integrated automatic startup folder purge hooks in `server.js` to ensure orphaned temporary code runs are deleted.
Intermittent Server Connection Errors Wildcard domains like sslip.io are frequently throttled or blocked by local ISP DNS servers, causing DNS lookups to time out on client browsers. Created a dedicated Elastic IP migration guide showing how to attach an Elastic IP in AWS to stabilize backend connections. Added fallback checking on Vite configurations.
Stale Contest Badges Contest statuses (Active/Completed) were stored as static string values in the database, causing expired contests to show as "LIVE NOW" to users. Rewrote the backend contest endpoints to compute contest statuses dynamically on request based on the current timestamp, automatically updating outdated records in the DB asynchronously.
Contest Workspace Escape Opening a problem from inside a contest redirected users to the generic problems page and wiped contest tracking headers. Modified client routers to pass contestId query parameters, keeping users inside the contest solver interface, and updated back arrow navigation to return users to the active contest dashboard.
